use std::sync::Arc;
let s = Arc::new(String::from("hello"));
let s_clone = Arc::clone(&s);
thread::spawn(move || println!("{}", s_clone));

这个为什么不报错：子线程可能比主线程活得久，会出现悬垂引用
没有想明白

我来用最直白、最戳痛点的方式给你讲透 ——为什么用了 Arc 就不报错了？
你现在卡住的点，我一句话先总结：
核心结论
因为 Arc 让数据不再属于主线程，而是 “谁都不独占、大家共享”。
编译器看不到 “主线程会先销毁数据” 的风险，所以允许子线程拿走。
1. 先回顾：普通引用为什么报错？
rust
运行
let s = String::from("hello");
thread::spawn(|| println!("{}", s)); // 报错
报错原因：
s 属于主线程
子线程借用了它
编译器无法保证主线程不会先结束、把 s 销毁
子线程就会变成悬垂引用（野指针）
所以 Rust 直接禁止。
2. 用了 Arc 为什么就不报错了？
rust
运行
let s = Arc::new(String::from("hello"));
let s_clone = Arc::clone(&s);
thread::spawn(move || println!("{}", s_clone));
这里发生了 3 个关键变化：
① Arc 把数据放到了堆上，不归任何线程拥有
数据不再属于主线程，而是属于 Arc 共享所有权。
② Arc::clone 不是复制数据，是复制 “共享指针”
每 clone 一次，引用计数 +1
计数 >0，数据绝对不会被销毁
③ move 把所有权移交给子线程
子线程现在拥有一个 Arc，而不是借用！
3. 编译器看到了什么？（最关键）
编译器看到：
子线程拿走的是 Arc 的所有权，不是借用
Arc 内部数据只有引用计数为 0 才会释放
只要子线程拿着 Arc，数据一定存活
所以编译器安心了，不报错。
4. 最通俗的比喻
普通变量 = 你家里的电视（只有你能销毁）
子线程 = 朋友来你家看电视（借用）
你可能突然把电视砸了 → 朋友就傻眼（悬垂引用）
Rust：禁止朋友来你家看电视 ❌
Arc = 把电视搬到公共广场
Arc::clone = 给朋友发一把广场钥匙
电视谁都不拥有，最后一个走的人负责关电视
你回家了，电视还在广场，朋友还能看 ✅
Rust：这很安全，不报错！
5. 一句话终极总结
普通线程报错，是因为子线程在借用主线程的数据。
Arc 不报错，是因为子线程拥有了共享所有权，数据不会提前销毁。
6. 你最关心的问题：为什么编译器不检查主线程先退出？
因为 Arc 内部用了原子引用计数，编译器信任它是安全的。
Rust 认为：
只要你持有 Arc，数据就一定活着
不需要再检查线程生命周期
这就是 Send + Sync 安全 的含义。
总结
普通引用：子线程借用 → 主线程可能先销毁 → 报错
Arc：子线程拥有共享所有权 → 数据不会提前销毁 → 安全
