Send，代表可以发送到任一线程中，但是Sync，不太理解，怎么就能保证Sync
用最简单的话讲懂 Send + Sync
这是 Rust 最核心、最容易懵的两个线程安全标记 trait，我用人话 + 例子一次性讲透：
先记住一句话（核心定义）
Send：这个类型的值，能安全地转移到另一个线程（所有权移交）
Sync：这个类型的引用（&T），能安全地在多个线程之间共享
1. Send：能 “搬家” 到别的线程
满足 Send = 你可以把这个变量从线程 A 完整搬到线程 B，不会出问题。
例子
i32、String、Vec、Box<T> 都是 Send
你可以把一个 String 传给子线程，这就是 Send
不是 Send：Rc<T>（引用计数非原子，跨线程会乱）
rust
运行
// String 是 Send，可以跨线程 move
let s = String::from("hello");
std::thread::spawn(move || {
    println!("{}", s); // 安全
});
2. Sync：能 “共享” 给别的线程（关键！）
满足 Sync = 多个线程同时只读访问 &T 是安全的。
最关键的一句话
T 是 Sync ⇔ &T 是 Send
意思是：
如果一个类型能安全地让多个线程同时只读共享，那它的引用就能安全发给别的线程。
Sync 到底怎么 “保证安全”？
Sync 不靠魔法，靠底层实现：
只有满足下面条件，才会实现 Sync：
多线程同时只读访问，不会产生数据竞争
内部没有需要线程独占的状态
如果有内部可变性，必须用原子操作 / 锁保护
哪些是 Sync？
i32：只读肯定安全 ✅
&i32：共享只读 ✅
Mutex<T>：内部有锁，多线程共享安全 ✅
Arc<T>：原子引用计数，共享安全 ✅
哪些不是 Sync？
Rc<T>：非原子计数，多线程共享会崩 ❌
&mut T：可变引用不能共享，天然不是 Sync ❌
3. 最直观的对比表
表格
类型	Send（能搬家）	Sync（能共享只读）	为什么
i32	✅	✅	简单值，无状态
String	✅	✅	所有权可转移，只读安全
Rc<T>	❌	❌	非原子计数，跨线程不安全
Arc<T>	✅	✅	原子计数
Mutex<T>	✅	✅	内部锁保护共享
&T	如果 T:Sync，则 &T:Send	—	这就是 Sync 的本质
4. 你最困惑的点：Sync 怎么保证安全？
安全不是靠关键字，是靠规则 + 底层实现：
Rust 不允许数据竞争
数据竞争 = 多线程同时至少一个写访问同一块内存
Sync 只允许 “只读共享”
&T 是只读的，天然不会产生数据竞争
如果要写，必须用锁
Mutex<T> 实现了 Sync，因为它用锁把写操作变成串行
所以：
Sync = 类型可以安全地在多线程里只读共享
Rust 编译器强制你遵守这个规则，不遵守就编译失败。
5. 终极极简总结
Send：我能搬到别的线程
Sync：我的只读引用能共享给别的线程
Sync 保证安全：因为只读共享 = 无数据竞争，编译器强制保证
总结
Send = 值可以转移到其他线程
Sync = 值的只读引用可以共享到其他线程
Sync 安全的本质：多线程只读访问不会产生数据竞争，Rust 编译期强制保障
你只要记住：Send 是转移，Sync 是共享只读，就完全懂了。
