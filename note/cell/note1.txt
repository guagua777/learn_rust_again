什么时候使用cell？

rust中Cell，一般什么时候用？感觉跟普通的变量也并没有什么区别，都是可以有多个读引用，一个写引用，没看出来有什么区别？
你这个直觉很正常 ——单线程、无共享时，Cell 确实和普通变量差不多；但它的真正价值是解决 Rust 的内部可变性（Interior Mutability），也就是：在不可变的东西里面，改它的内容。下面分三部分讲清楚：
一、先纠正一个误区
你说 “普通变量也是多个读、一个写”——不对：
普通变量（继承可变性）
let x: i32：完全不可变，不能写
let mut x: i32：必须独占（&mut）才能写；一旦有 &x，就不能 &mut x
Cell（内部可变性）
let c = Cell::new(5)：c 本身是不可变（&Cell），但可以直接 c.set (10)
核心：用 &Cell 就能改里面的值，不需要 &mut
一句话：
普通变量：不可变绑定 → 完全不能改
Cell：不可变绑定 → 照样能改内部值
二、什么时候必须用 Cell？
1. 不可变结构体里，要改某个字段（最常见）
Rust 默认：结构体不可变 → 所有字段都不可变。
用 Cell 可以 “局部可变”：
rust
运行
use std::cell::Cell;

struct Foo {
    // 普通字段：继承可变性
    a: i32,
    // Cell 字段：内部可变性
    b: Cell<i32>,
}

fn main() {
    // foo 是不可变的！
    let foo = Foo {
        a: 1,
        b: Cell::new(2),
    };

    // 错误：foo 不可变，a 不能改
    // foo.a = 10;

    // 正确：b 是 Cell，&foo 就能 set
    foo.b.set(20);
    println!("b = {}", foo.b.get()); // 20
}
场景：结构体整体逻辑上不可变，但个别字段（如缓存、计数器、状态标记）需要频繁修改。
2. 共享引用（&T）需要修改数据（Rc 必备）
Rc（共享所有权）只能给 &T，不能给 &mut T——没有 Cell 就完全改不了：
rust
运行
use std::rc::Rc;
use std::cell::Cell;

fn main() {
    // 共享一个 Cell
    let shared = Rc::new(Cell::new(0));

    let c1 = shared.clone();
    let c2 = shared.clone();

    // 两个不可变引用，都能改里面的值
    c1.set(1);
    c2.set(2);

    println!("{}", shared.get()); // 2
}
场景：多所有者、共享状态（如事件回调、状态机、组件通信）。
3. 逻辑不可变的方法里，要改内部状态
比如一个 “只读” 的对象，但内部需要记录访问次数、缓存结果：
rust
运行
use std::cell::Cell;

struct Cache {
    value: i32,
    hits: Cell<u32>,
}

impl Cache {
    // 方法是 &self（逻辑不可变）
    fn get(&self) -> i32 {
        // 但可以改 hits
        self.hits.set(self.hits.get() + 1);
        self.value
    }
}
三、Cell 和普通变量的本质区别（为什么不一样）
表格
维度	普通变量（i32）	Cell<i32>
可变性来源	绑定是否 mut（继承）	内部封装（内部可变性）
修改条件	必须 &mut T	只要 &Cell<T> 就能 set
共享时修改	❌ 有 & 就不能 &mut	✅ 多个 &Cell 都能改
安全保证	静态借用检查	运行时无检查（靠值拷贝保证安全）
适用类型	所有类型	只能是 Copy 类型（i32、bool、f64…）
关键差异一句话：
普通变量：共享即不可改
Cell：共享也能改（仅限 Copy 类型）
四、总结：什么时候用 Cell？
不可变容器（结构体、Rc）里要改数据；
多个共享引用（&）需要修改同一个值；
逻辑不可变的方法里要改内部状态；
类型是 Copy（数字、bool、简单枚举）—— 非 Copy 用 RefCell。
一句话：当你想 “不可变的外表，可变的内心” 时，用 Cell。
